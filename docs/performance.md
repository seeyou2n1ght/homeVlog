# HomeVlog 性能分析与调优 (Benchmark)

本中心记录了项目在不同硬件配置下的表现，并提供了针对性的调优建议。

## 1. 性能 Benchmark (2026-04-24)

**测试环境：**
- **CPU**: Intel i5-12600K
- **GPU**: NVIDIA RTX 3060 Ti (8GB)
- **Input**: 4K HEVC (H.265) @ 30fps
- **Config**: 640x640 @ 5fps sampling

### 1.1 异构模式对比

| 架构版本 | 硬件分配 | 综合吞吐量 (System FPS) | GPU 利用率 | 瓶颈点 |
| :--- | :--- | :--- | :--- | :--- |
| V2.3 (早期) | CUDA 解码 + CUDA 推理 | ~55 fps | 波动大 (10%-90%) | 上下文切换 / 显存带宽 |
| **V2.4 (当前)** | **QSV 解码 + CUDA 推理** | **~125 fps** (147 fps Peak) | **稳健 (40% 推理负载)** | **核显解码带宽** |

**分析**：将解码压力从独显移至核显后，独显能够以完美的 Batch 模式运行，推理延时极低且显存占用极其稳定。

## 2. 资源占用透视

在处理 4K 视频流时，各组件负载表现：
- **Intel UHD 770 (核显)**: **95% - 100%**。正在全速进行 4K HEVC 硬解与降采样。
- **RTX 3060 Ti (独显)**: **~40%**。推理引擎处于“喂不饱”状态，等待核显产帧。
- **CPU**: **30% - 45%**。主要消耗在 Python 线程调度与 FFmpeg 子进程管理上。

## 3. 调优建议 (Tuning Tips)

### 3.1 增加吞吐量
如果你的独显利用率低于 50%，可以通过以下方式提升：
- **调整 Sampling FPS**: 如果监控场景对实时性要求不高，将 `config.yaml` 中的 `fps` 降低（如 3fps），可以减少解码压力。
- **多实例运行**: 如果显存充足，可以考虑启动两组进程分别处理不同的文件子集。

### 3.2 降低延迟
- **Batch Size**: 目前 `batch_size=12` 是性价比最高的平衡点。如果显存极小（如 4G），建议降至 4 或 8。
- **Engine 精度**: 始终使用 `FP16` 模式。虽然 `INT8` 理论上更快，但对于安防场景的 YOLO 模型，FP16 在速度和精度间平衡最好。

### 3.3 解决 I/O 瓶颈
- 确保输出目录在 **SSD** 上。虽然切割是 `copy` 模式，但在并发 4 线程切割时，机械硬盘（HDD）的寻道延迟会严重拖慢整个流水线的最后环节。
